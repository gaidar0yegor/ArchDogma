# Catalog of Dogmas

> Священные коровы программирования — и реальные кейсы, когда они убивали проекты.

> ⚠️ **Статус каталога: DRAFT / зачаточное состояние.**
> Большинство догм помечены `[NEED POSTMORTEMS]` и `[TO BE WRITTEN]`. Это не баг — это честное состояние на старте. Мы ищем кейсы, не выдумываем их.
>
> **v0.1 scope:** заполняем полностью только три догмы, помеченные 🎯 — **Microservices**, **TDD**, **DRY**. Остальные — бэклог.
>
> Контрибьюторы welcome — особенно с реальными ссылками на пост-мортемы.

**Правила каталога:**

1. Каждый кейс — со ссылкой на реальный источник (пост-мортем, блог-пост, конф-доклад, тред). Без ссылки — не кейс.
2. Никаких выдуманных чисел. Если не знаем — пишем `[NEED POSTMORTEMS]`. Не пишем «обычно на 55% coverage команда ломается» — это гороскоп.
3. Догма не враг. Догма становится врагом при определённых параметрах. Наша работа — найти эти параметры, а не проповедовать «догма плохая».
4. Честный вывод в конце каждой догмы. Не «it depends», а что-то операциональное.

Формат каждой записи:

```
## [Название догмы]

**Что говорит:** короткое определение
**Откуда пришло:** книга / человек / движение
**При каких параметрах начинает душить:** список условий
**Реальные кейсы краха:** [ссылки или NEED POSTMORTEMS]
**Реальные кейсы где помогла:** [ссылки или NEED DATA]
**Контр-догмы:** зрелые возражения с атрибуцией и ссылками
**Honest verdict:** когда следовать, когда нарушать
```

Контр-догма — это не «противоположная догма» (типа «всегда копипасть»). Это зрелая критика с именем, источником и областью применимости. Без атрибуции — не контр-догма, а чьё-то мнение.

## 1. 100% Test Coverage

**Что говорит:** Всё должно быть покрыто unit-тестами. Непокрытый код — это риск.

**Откуда пришло:** TDD-движение, Kent Beck, Uncle Bob, XP.

**При каких параметрах начинает душить:**
- Маленькая команда + жёсткий дедлайн + много legacy
- Исследовательский код / прототипы
- Код с сильной привязкой к внешнему миру (UI, hardware, интеграции)
- Когда покрытие становится KPI, а не средством

**Реальные кейсы краха:** `[NEED POSTMORTEMS]` — ищем истории про то, как 100% coverage превратился в ritual-тесты, которые ничего не ловили.

**Реальные кейсы где помогла:** `[NEED DATA]`

**Honest verdict:** `[TO BE WRITTEN после сбора кейсов]`

**Что говорит:** Разделяй код на слои (domain, application, infrastructure, presentation). Зависимости идут внутрь.

- Когда добавление поля в БД требует правок в 7 файлах

**Реальные кейсы краха:** `[NEED POSTMORTEMS]` — ищем разборы стартапов, которые утонули в абстракциях.

## 3. 🎯 DRY (Don't Repeat Yourself) — v0.1 priority

**Что говорит:** Никогда не копипасть. Любое повторение — кандидат на абстракцию.

**Откуда пришло:** «The Pragmatic Programmer», Hunt & Thomas, 1999.

**При каких параметрах начинает душить:**
- Когда две похожие вещи объединяются в одну абстракцию, а потом расходятся
- Shared libraries между командами с разными release-циклами
- Слишком ранняя абстракция (до 3-го реального use case)

**Реальные кейсы краха:** `[NEED POSTMORTEMS]` — особенно интересны истории про shared monolith, который никто не может тронуть.

**Реальные кейсы где помогла:** `[NEED DATA]`

**Контр-догмы:**
- **WET** (Write Everything Twice) — фолк-контр-догма, автор анонимен. Область: пока не увидел повторение дважды — не абстрагируй.
- **Rule of Three** — Don Roberts, приведено в Martin Fowler, «Refactoring» (1999). Правило: три повторения — только тогда кандидат на абстракцию.
- **The Wrong Abstraction** — Sandi Metz, 2016. Тезис: «Duplication is far cheaper than the wrong abstraction». Источник: [sandimetz.com/blog/2016/1/20/the-wrong-abstraction](https://sandimetz.com/blog/2016/1/20/the-wrong-abstraction).
- **AHA** (Avoid Hasty Abstractions) — Kent C. Dodds, 2019. Источник: [kentcdodds.com/blog/aha-programming](https://kentcdodds.com/blog/aha-programming). *Примечание: раньше в каталоге AHA ошибочно атрибутировался Sandi Metz. Это разные авторы — Metz про «wrong abstraction», Dodds про «avoid hasty».*

**Honest verdict (v0.1 draft, ждёт кейсов):**

Следуй DRY, когда:
- Повторяется **знание** (бизнес-правило, формула, инвариант), а не форма кода. Дубликат правила «НДС = 20%» в трёх местах — это баг, не DRY-нарушение.
- Ты видишь третье повторение (Rule of Three), и все три вызываются из одного контекста / одной командой / одного релизного цикла.
- Цена ошибочной абстракции ниже цены дубля — например, стоимость расхождения трёх мест выше, чем стоимость открутить абстракцию назад.

Нарушай DRY, когда:
- Два куска кода **выглядят одинаково, но меняются по разным причинам** (разные стейкхолдеры, разные релиз-циклы, разные домены). Это не дубль — это совпадение формы. Объединишь — склеишь две независимые истории.
- Абстракция пересекает границу команд или сервисов. Shared library между тремя командами с разными релиз-циклами — почти всегда медленное самоубийство. Копипаст локальнее и честнее.
- Есть только 1–2 повторения, и ты ещё не видел, как оно реально меняется. Ждать третьего использования дешевле, чем откатывать абстракцию.
- Код — исследовательский / одноразовый. DRY в spike-коде добавляет связность, которую всё равно выбросишь.

Главный сигнал, что ты промазал с абстракцией: **каждое новое требование добавляет в «общую» функцию `if`-флаг**. Это не расширение — это признание, что ты склеил две разные вещи. Разлепляй обратно, пока не поздно.

## 4. 🎯 Microservices для всего — v0.1 priority

**Что говорит:** Монолит — зло. Режь систему на маленькие сервисы.

**Откуда пришло:** Netflix, Amazon, ThoughtWorks, ~2014.

**При каких параметрах начинает душить:**
- Команда < 10 человек
- Один продукт, не независимые бизнес-юниты
- Distributed transactions становятся нормой
- Debug занимает 3 часа, потому что trace разлетается по 12 сервисам

**Реальные кейсы краха:** `[NEED POSTMORTEMS]` — отличные кандидаты: Istio retrospectives, Segment's «Goodbye Microservices», Prime Video monolith migration.

**Реальные кейсы где помогла:** Netflix, Amazon (на определённом масштабе). Нужны ссылки.

**Контр-догмы:**
- **MonolithFirst** — Martin Fowler, 2015. Тезис: почти все успешные микросервисные системы начинались как монолиты и были распилены позже. Начинать с микросервисов — ошибка. Источник: [martinfowler.com/bliki/MonolithFirst.html](https://martinfowler.com/bliki/MonolithFirst.html).
- **Modular Monolith** — подход, систематизированный Shopify и другими. Модули с чёткими границами внутри одного процесса. Без сетевых границ, но с возможностью разделить позже. Нужны ссылки на inside-Shopify пост.
- **Prime Video monolith migration** — Amazon Prime Video Tech Blog, 2023. История перехода от serverless/микросервисов обратно к монолиту со снижением инфраструктурных затрат на 90%. Источник: [primevideotech.com/video-streaming/scaling-up-the-prime-video-audio-video-monitoring-service-and-reducing-costs-by-90](https://www.primevideotech.com/video-streaming/scaling-up-the-prime-video-audio-video-monitoring-service-and-reducing-costs-by-90).

**Honest verdict (v0.1 draft, ждёт кейсов):**

Следуй подходу микросервисов, когда:
- У тебя действительно несколько **независимых бизнес-доменов** с разными командами, релизными циклами и масштабом.
- Команда больше 10–15 человек и уже есть конкретная боль от монолита (конфликты при деплое, один медленный тест блокирует весь CI, релизы бутылочным горлышком).
- Ты готов платить операционную цену распределённой системы: network latency, distributed tracing, eventual consistency, отдельный мониторинг на каждый сервис.

Нарушай (и лучше не начинай с микросервисов), когда:
- Команда < 10 человек и/или один продукт.
- Проект в фазе стартапа / MVP — домен ещё не устоялся.
- Появляются distributed transactions или debug-сессии, где trace разлетается по 5+ сервисам.
- Основная боль — в **сложности бизнес-логики**, а не в масштабе. Микросервисы сложность не убирают, они её **перемещают** в сеть.

Главный сигнал, что ты промазал: **distributed monolith**. Куча мелких сервисов, которые всё равно меняются вместе и деплоятся вместе, но теперь ещё через сеть и с распределённым failure mode. Если релиз одного сервиса в 80% случаев требует релиза второго — у тебя не микросервисы, у тебя монолит по сетке.

Правило на практике: **начинай с монолита (или modular monolith)**. Выделяй микросервис только когда он решает конкретную измеримую боль, а не «на будущее» и не «потому что Netflix». См. контр-догмы выше — Fowler «MonolithFirst», Prime Video 2023 (возврат к монолиту = минус 90% инфраструктуры).

## 5. OOP как единственная истина (наследование везде)

## 6. 🎯 TDD (Test-Driven Development) — сначала тест — v0.1 priority

**Что говорит:** Red-Green-Refactor. Сначала пишем тест, потом код.

**Откуда пришло:** Kent Beck.

**При каких параметрах начинает душить:**
- Неизвестная область — ещё не знаешь, каким должен быть API
- UI-код, визуализации
- Research / exploratory coding

**Реальные кейсы краха:** `[NEED POSTMORTEMS]` — включая знаменитый пост DHH «TDD is dead. Long live testing».

**Реальные кейсы где помогла:** `[NEED DATA]`

**Контр-догмы:**
- **Spike First** — Kent Beck, часть оригинального XP. Сам автор TDD допускает: в неизвестной области сначала делается spike (одноразовый прототип без тестов), потом выбрасывается, потом пишется заново через TDD. Источник: Kent Beck, «Extreme Programming Explained» (1999).
- **Test After** / **Test Last** — фолк-контр-практика. Тесты пишутся после реализации, когда API уже устоялся. Область: UI, exploratory work, research code.
- **Characterization Testing** — Michael Feathers, «Working Effectively with Legacy Code» (2004). Для legacy без тестов: пишешь тест, который фиксирует *текущее* поведение (даже если оно кривое), чтобы можно было рефакторить безопасно. Это не TDD — обратное направление.
- **«TDD is dead. Long live testing.»** — DHH (David Heinemeier Hansson), 2014. Тезис: TDD как идеология привела к over-mocking и дизайну под тесты вместо дизайна под задачу. Источник: [dhh.dk/2014/tdd-is-dead-long-live-testing.html](https://dhh.dk/2014/tdd-is-dead-long-live-testing.html). Контекст: последовала серия дебатов «Is TDD Dead?» с Kent Beck и Martin Fowler.

**Honest verdict (v0.1 draft, ждёт кейсов):**

Следуй классическому Red-Green-Refactor, когда:
- Ты уже понимаешь домен и примерно знаешь, каким должен быть API. Тест пишется как **спецификация**, не как исследование.
- Работаешь над стабильной бизнес-логикой, которую планируешь рефакторить (тесты = сеть безопасности).
- Цена ошибки в проде очень высокая (финансы, безопасность, медицина, биллинг).

Нарушай и делай иначе, когда:
- Ты в неизвестной области — новая фича, новый домен, research. Используй **Spike First** (Kent Beck, XP): грязный прототип без тестов, чтобы понять, *что именно* нужно построить. Потом выбрасываешь и переписываешь под тестами.
- Работаешь с UI, сложными визуальными состояниями или внешними интеграциями. API устаканивается только после нескольких итераций — лучше **Test After**, когда форма контракта стала понятной.
- Работаешь с legacy без тестов. Классический TDD тут невозможен по определению. Используй **Characterization Testing** (Michael Feathers): фиксируешь *текущее* поведение, даже если оно кривое, чтобы рефакторить безопасно.
- Прототип / proof of concept, который будет выброшен через неделю. Тесты здесь — overhead без возврата.

Главный сигнал, что ты промазал: **ты пишешь тест раньше, чем понял, что код должен делать**. Результат — хорошо протестированный неправильный дизайн + лес моков, которые повторяют структуру кода, а не проверяют поведение. Тесты становятся копией имплементации и ломаются при каждом рефакторинге. Это не сеть безопасности, это гирь на ногах.

TDD — инструмент, не религия. Нормальные тесты, написанные чуть позже — лучше, чем «правильные» тесты, написанные с первой строки вокруг несуществующего понимания задачи.

## 7. SOLID как закон

**Что говорит:** SRP, OCP, LSP, ISP, DIP — пять принципов, которым должен следовать весь код.

**Откуда пришло:** Uncle Bob, 2000-е.

**При каких параметрах начинает душить:**
- Когда Single Responsibility превращается в 1 функцию = 1 класс
- Когда Dependency Inversion приводит к интерфейсам с одной реализацией
- Когда Open/Closed заставляет добавлять extension points, которые никто никогда не использует

## 8. Self-documenting code (комментарии не нужны)

**Что говорит:** Хороший код читается без комментариев. Комментарии — признак непонятного кода.

## 9. Premature optimization is the root of all evil

**Что говорит:** Не оптимизируй, пока не профилировал. (Обычно цитируется без контекста.)

**Откуда пришло:** Donald Knuth, 1974. Полная цитата: «We should forget about small efficiencies, say about 97% of the time: premature optimization is the root of all evil. Yet we should not pass up our opportunities in that critical 3%.»

**При каких параметрах начинает душить (в искажённой форме):**
- Когда превращается в «никогда не оптимизируй»
- Когда архитектурные решения о производительности откладываются до «когда понадобится» и становится поздно
- Data pipelines, real-time системы, embedded

**Реальные кейсы краха:** `[NEED POSTMORTEMS]`

**Honest verdict:** `[TO BE WRITTEN]`

Проблема здесь — не Knuth, а обрезанная цитата.

## 10. Functional purity / Immutability везде

**Что говорит:** Избегай мутаций. Чистые функции. Никаких side effects.

**Откуда пришло:** Haskell-сообщество, FP-ренессанс 2010-х, React/Redux.

**При каких параметрах начинает душить:**
- Высокопроизводительные пайплайны (копирование vs in-place)
- Большие структуры данных в памяти
- Когда persistent data structures добавляют 3x overhead без причины

**Реальные кейсы краха:** `[NEED POSTMORTEMS]`

**Реальные кейсы где помогла:** `[NEED DATA]`

## Догмы-кандидаты (ещё не разобраны)

- «YAGNI» — You Aren't Gonna Need It
- «KISS» — Keep It Simple, Stupid
- «Agile» и его ритуалы
- «Конфигурация вместо кода»
- «Serverless для всего»
- «GraphQL вместо REST»
- «NoSQL вместо SQL»
- «Event Sourcing»
- «Domain-Driven Design как обязательная практика»
- «Code reviews должны быть обязательными всегда»
- «Feature flags вместо веток»
- «Монорепо vs полирепо»
- «Pair programming всегда»
- **God File / God Class** — антипаттерн-*последствие*, не догма. Появляется от страха рефакторинга + отсутствия тестов + дедлайнов, а не от избытка дисциплины. Реальные источники с ссылками: SQLite amalgamation как *осознанное* решение для производительности ([sqlite.org/amalgamation.html](https://sqlite.org/amalgamation.html), ~238k строк), Knight Capital 2012 — $440M убытка от dead code в legacy ([SEC 34-70694](https://www.sec.gov/litigation/admin/2013/34-70694.pdf)), Michael Feathers «Working Effectively with Legacy Code» (2004) — каноническое описание God Class. Публичные пост-мортемы с конкретными проектами из банков/страхования/авиации — `[NEED POSTMORTEMS]`.

Эти — в бэклог. Сначала разбираем первую десятку до уровня, где у каждой есть 2–3 реальных кейса с ссылками.

## Как контрибьютить в каталог

1. Нашёл реальный пост-мортем / разбор / блог-пост, где догма убила проект? → Issue с ссылкой + краткий summary.
2. Работал в команде, где догма душила, и можешь рассказать честно (с деталями, не «вроде бы кажется»)? → Issue с меткой `personal-case`. Мы не публикуем имена без разрешения.
3. Видишь в каталоге утверждение без ссылки? → Issue с меткой `honesty-bug`.

**Запрещено:**
- Выдумывать цифры и «обычно» без источника
- Писать «на 4-м спринте команда начинает...» без референса на реальную команду
- Использовать AI для генерации «кейсов». Каталог — про реальный опыт реальных людей.
